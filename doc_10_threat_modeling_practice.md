# Threat Modeling Practice

威胁建模不是安全团队在评审会上挑毛病，而是让设计阶段提前暴露风险和假设。它的价值在于减少后期返工，并让工程团队清楚知道哪些安全属性不能被破坏。

## 1. 什么时候做

建议在这些节点做威胁建模：

- 新系统设计阶段。
- 架构边界变化。
- 引入新的数据类型。
- 增加第三方集成。
- 权限模型变化。
- 上线前做轻量复核。
- 重大安全事件后的架构复盘。

越晚做，越容易变成补丁清单。

## 2. 输入材料

一次有效威胁建模需要：

| 材料 | 目的 |
| --- | --- |
| 架构图 | 理解组件和边界 |
| 数据流图 | 识别敏感数据路径 |
| 资产清单 | 明确保护对象 |
| 身份与权限模型 | 识别越权风险 |
| 第三方依赖 | 识别供应链和信任边界 |
| 日志与监控设计 | 判断可检测性 |
| 业务滥用场景 | 识别非技术攻击 |

没有数据流图时，威胁建模很容易只讨论组件而忽略真正的数据风险。

## 3. 信任边界

重点标出这些边界：

- 互联网到服务入口。
- 用户到管理接口。
- 前端到后端 API。
- 应用到数据库。
- 内部服务之间。
- CI/CD 到生产环境。
- 企业环境到第三方 SaaS。
- AI 检索内容到工具调用。

每跨过一个信任边界，都应该问：身份如何确认，输入如何校验，权限如何限制，日志如何记录。

## 4. STRIDE 简化版

| 类别 | 问题 |
| --- | --- |
| Spoofing | 攻击者能不能伪装成别人 |
| Tampering | 数据或请求能不能被篡改 |
| Repudiation | 行为是否可追责 |
| Information disclosure | 敏感信息是否会泄露 |
| Denial of service | 系统是否容易被打垮 |
| Elevation of privilege | 是否能越权 |

不需要机械套模板。模板是为了启发问题，不是为了完成表格。

## 5. 风险描述格式

建议统一写法：

```text
如果 [攻击者能力]，在 [前提条件] 下，通过 [攻击路径]，可能导致 [影响]。
```

示例：

```text
如果普通用户能控制导入文件内容，在服务端解析器未限制资源消耗的情况下，通过构造异常大文件，可能导致任务队列阻塞并影响其他租户。
```

清晰的风险描述能直接转化为控制措施和测试用例。

## 6. 控制措施

常见控制类型：

| 控制 | 作用 |
| --- | --- |
| Preventive | 阻止风险发生，例如鉴权、输入校验 |
| Detective | 发现风险，例如日志、告警 |
| Corrective | 修复影响，例如回滚、隔离 |
| Compensating | 在无法直接修复时降低风险 |

不要把所有风险都交给检测。能在设计层面消除的风险，优先消除。

## 7. 输出物

威胁建模输出应该短而有用：

- 架构图和数据流图。
- 关键资产和信任边界。
- 风险列表和优先级。
- 已有控制措施。
- 待补控制措施。
- owner 和截止日期。
- 需要安全测试验证的点。

输出物的目标是推动工程动作，不是写一份没人维护的报告。

## 8. 优先级判断

风险优先级建议综合：

- 资产敏感度。
- 攻击前提是否容易满足。
- 影响范围。
- 是否可被检测。
- 是否已有补偿控制。
- 修复成本。

高影响、低前提、不可检测的风险优先处理。

## 9. 常见失败模式

| 问题 | 结果 |
| --- | --- |
| 只让安全团队参加 | 不了解真实业务和架构 |
| 只找漏洞不讲控制 | 工程团队难以落地 |
| 风险描述太抽象 | 无法转成需求 |
| 没有 owner | 问题长期悬空 |
| 不复盘历史事件 | 重复踩坑 |

威胁建模要成为工程流程的一部分，而不是上线前临时加的一道门。

## 10. 轻量模板

```text
系统:
资产:
数据流:
信任边界:
攻击者:
风险:
现有控制:
缺口:
建议:
owner:
验证方式:
```

好的威胁建模会让设计更清楚，也会让安全要求更容易被工程接受。
