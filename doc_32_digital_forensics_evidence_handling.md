# Digital Forensics Evidence Handling

数字取证的重点是保持证据可信。即使没有进入法律诉讼，证据完整性、来源、时间线和处理记录也会直接影响事件复盘和根因判断。

## 1. 证据原则

| 原则 | 说明 |
| --- | --- |
| Integrity | 证据未被篡改 |
| Chain of custody | 处理链路可追踪 |
| Repeatability | 分析过程可复现 |
| Minimal alteration | 采集尽量减少对原始系统影响 |
| Context | 证据有来源、时间和环境说明 |

证据处理越规范，后续结论越站得住。

## 2. 证据类型

常见证据：

- 磁盘镜像。
- 内存镜像。
- 日志文件。
- 云审计事件。
- 网络流量。
- 进程和连接快照。
- 可疑文件样本。
- 邮件原文。
- 配置快照。

不同证据有不同易失性，内存和运行状态通常优先级更高。

## 3. 采集记录

每次采集记录：

```text
case_id
collector
time
source
method
tool_version
hash
storage_location
notes
```

没有采集记录的证据，后续解释成本会很高。

## 4. 完整性校验

对关键证据计算 hash：

- 采集时计算。
- 传输后校验。
- 分析前校验。
- 归档前校验。

hash 不能证明证据真实，但能证明采集后未变化。

## 5. 保存和访问

要求：

- 证据存储在受控位置。
- 访问权限最小化。
- 下载和查看有日志。
- 原始证据只读保存。
- 分析副本和原始证据分离。
- 保留期符合事件和合规要求。

不要在共享网盘随意传递高敏证据。

## 6. 时间线

时间线应统一：

- 时区。
- 时间精度。
- 事件来源。
- 是否为发生时间或采集时间。
- 日志延迟。

时间线分析要标记不确定性，避免把延迟日志当成真实发生顺序。

## 7. 云取证特点

云环境要关注：

- 控制面日志。
- 资源快照。
- 临时凭证。
- 自动化执行记录。
- 跨区域和跨账号复制。
- 日志保留和删除权限。

云证据常来自 API 和托管服务，不一定能获取传统磁盘镜像。

## 8. 输出报告

报告应包含：

- 结论摘要。
- 证据来源。
- 时间线。
- 影响范围。
- 根因判断。
- 不确定点。
- 后续建议。

取证报告要区分事实、推断和假设。

## 9. 指标

| 指标 | 目标 |
| --- | --- |
| 证据采集模板覆盖 | 100% |
| hash 校验覆盖 | 关键证据 100% |
| 证据访问审计 | 100% |
| 时间线完整度 | 持续提升 |
| 取证工具版本记录 | 100% |

取证要让事件判断有可信依据，保存文件只是手段。
