# Log Investigation Playbook

日志调查需要方法，随机关键词搜索效率很低。好的调查流程能快速缩小范围、建立时间线、验证假设，并把发现转化为响应动作。

## 1. 调查起点

先明确：

- 告警是什么。
- 触发实体是谁。
- 事件时间。
- 影响资源。
- 规则逻辑。
- 告警置信度。
- 是否有类似历史事件。

不理解告警逻辑就开始调查，容易被表面字段误导。

## 2. 时间窗口

建议：

- 告警前 24 小时看初始访问。
- 告警前后 1 小时看直接行为。
- 告警后 24 小时看扩散和影响。
- 高级事件按周扩展。

时间窗口要根据攻击链调整，固定模板只能做起点。

## 3. 关联维度

常用 pivot：

| 维度 | 用途 |
| --- | --- |
| user_id | 查身份活动 |
| device_id | 查终端行为 |
| source_ip | 查来源和其他账号 |
| session_id | 查同一会话动作 |
| resource_id | 查资源访问 |
| request_id | 查跨服务请求 |
| token_id | 查凭证使用 |

调查价值来自关联，单条日志只能提供线索。

## 4. 假设驱动

示例假设：

```text
账号可能被接管。
服务账号可能被滥用。
数据可能被批量导出。
云密钥可能泄露。
终端可能执行了恶意脚本。
```

每个假设都要对应可验证证据。

## 5. 证据分级

| 证据 | 强度 |
| --- | --- |
| 单次失败登录 | 弱 |
| 异常登录后权限变更 | 中 |
| 异常登录后创建 token 并访问敏感数据 | 强 |
| 多数据源一致指向同一行为 | 很强 |

不要把单个异常直接当成恶意结论。

## 6. 时间线输出

时间线建议格式：

```text
time
source
actor
action
resource
evidence_link
assessment
```

时间线中应区分事实和推断。

## 7. 常见调查路径

账号异常：

- 登录来源。
- MFA 事件。
- 会话和 token。
- 权限变更。
- 数据访问。
- 邮箱规则和 OAuth 授权。

云异常：

- API 调用者。
- IAM 变更。
- 新资源。
- 网络和存储变更。
- 凭证使用范围。

终端异常：

- 进程树。
- 命令行。
- 网络连接。
- 文件写入。
- 用户登录。

## 8. 结论写法

结论应包含：

- 已确认事实。
- 影响范围。
- 仍不确定的点。
- 建议响应动作。
- 需要补充的数据。

避免只写“疑似异常”。要说明为什么疑似、还缺什么证据。

## 9. 指标

| 指标 | 目标 |
| --- | --- |
| 平均调查时间 | 持续降低 |
| 缺失字段次数 | 持续下降 |
| 多源关联覆盖 | 持续提升 |
| 误判复盘数量 | 用于改进规则 |
| 调查模板使用率 | 持续提升 |

日志调查的本质是用证据排除不可能，逐步收敛到可执行判断。
