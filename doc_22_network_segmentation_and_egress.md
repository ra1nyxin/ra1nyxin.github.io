# Network Segmentation and Egress Control

网络安全的重点已经从“内外网边界”转向“访问关系控制”。内部网络不应被默认信任，出口访问也不应完全开放。分段和出口控制能显著降低横向移动、数据外传和命令控制风险。

## 1. 分段目标

| 目标 | 说明 |
| --- | --- |
| Reduce blast radius | 限制单点失陷影响范围 |
| Protect crown jewels | 保护关键系统和数据 |
| Control admin paths | 管理面访问必须受控 |
| Limit lateral movement | 内部服务不能随意互通 |
| Improve visibility | 让访问关系可审计 |

分段不是为了画复杂网络图，而是减少不必要的连通性。

## 2. 访问关系建模

先回答：

- 谁需要访问谁。
- 使用什么协议和端口。
- 是否需要双向通信。
- 访问频率是否稳定。
- 是否包含管理权限。
- 是否跨环境或跨租户。

没有访问关系模型，防火墙规则会变成历史遗留集合。

## 3. 分段边界

优先控制：

- 用户办公网到生产环境。
- CI/CD 到生产管理面。
- 应用层到数据库层。
- 普通服务到身份和密钥系统。
- 第三方网络到内部资源。
- 开发测试环境到生产环境。

开发环境不应默认能访问生产数据。

## 4. 出口控制

出口控制关注系统能连到哪里。

建议：

- 默认阻断未知外联。
- 对业务必须域名建立 allowlist。
- 对云 API、包管理器、Webhook 做独立策略。
- 记录 DNS 和代理日志。
- 对大流量上传做告警。
- 对新域名和低信誉域名提高风险。

出口控制要考虑开发体验，可以先监控后阻断。

## 5. DNS 可观测性

DNS 是发现外联的重要信号。

可检测：

- 新注册域名访问。
- 随机子域名高频查询。
- 内部主机查询动态 DNS。
- 非预期进程触发外联。
- 查询量突然上升。

DNS 日志要能关联主机、用户和应用，否则调查价值会下降。

## 6. 管理面隔离

管理面包括：

- SSH、RDP、WinRM。
- Kubernetes API。
- 云控制台和 API。
- 数据库管理端口。
- CI/CD 管理接口。
- 监控和日志平台。

管理面应通过堡垒、强认证、JIT 权限和审计访问，不应该对办公网大范围开放。

## 7. 规则治理

网络规则需要 owner。

每条规则应包含：

- 来源。
- 目标。
- 协议和端口。
- 业务理由。
- owner。
- 创建时间。
- 过期时间。
- 最近命中时间。

长期无命中的规则应进入清理流程。

## 8. 检测场景

优先检测：

- 办公网主机访问生产数据库端口。
- 非 CI/CD 主体访问发布系统。
- 生产主机访问新外部域名。
- 内部主机扫描多个网段。
- 数据库服务器发起异常公网连接。
- 规则变更后出现新访问路径。

网络检测要结合资产重要性，不是所有连接异常都有同等价值。

## 9. 指标

| 指标 | 目标 |
| --- | --- |
| 管理面暴露路径 | 持续减少 |
| 无 owner 网络规则 | 0 |
| 无命中规则比例 | 持续下降 |
| 默认允许区域 | 持续减少 |
| 出口 allowlist 覆盖 | 关键系统优先 |
| 异常外联调查时间 | 持续降低 |

有效的网络分段会让攻击者每前进一步都遇到新的授权边界。
