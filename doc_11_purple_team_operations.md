# Purple Team Operations

紫队不是红队和蓝队坐在一起开会，而是把攻击模拟、检测验证、响应改进和控制加固串成闭环。它的目标不是证明谁更强，而是让组织知道哪些攻击路径能被发现、能被阻断、能被恢复。

## 1. 紫队目标

| 目标 | 说明 |
| --- | --- |
| Validate controls | 验证安全控制是否按预期工作 |
| Improve detection | 改进检测覆盖和告警质量 |
| Exercise response | 演练调查和遏制流程 |
| Reduce assumptions | 用证据替代口头信心 |
| Build shared language | 让红蓝双方用同一套技术语义沟通 |

紫队成功的标志不是发现很多问题，而是问题能被快速转化为工程改进。

## 2. 选择场景

场景应来自真实风险：

- 高权限账号被钓鱼。
- 云密钥泄露。
- 代码仓库 token 被滥用。
- CI/CD 发布链路被篡改。
- SaaS 数据被批量导出。
- 终端执行异常脚本。
- 生产环境出现横向访问。

不要为了覆盖技术清单而做演练。优先选择业务影响明确、检测价值高的路径。

## 3. 演练结构

一次紫队活动建议分为：

| 阶段 | 内容 |
| --- | --- |
| Planning | 明确目标、范围、成功标准和安全边界 |
| Emulation | 执行受控攻击模拟 |
| Detection review | 检查是否产生遥测和告警 |
| Response review | 验证分析和处置流程 |
| Gap analysis | 记录控制缺口 |
| Engineering follow-up | 修复规则、日志、权限和流程 |
| Retest | 重新验证 |

没有 retest 的紫队活动，只能算一次安全观察。

## 4. 成功标准

每个场景都要提前定义成功标准。

示例：

```text
当一个非交互式服务账号在非预期地区调用高权限 API 时：
1. 云审计日志能记录 actor、source_ip、action、resource。
2. 检测规则在 10 分钟内触发。
3. 告警能说明账号 owner 和最近变更。
4. 响应流程能撤销凭证并确认影响范围。
```

成功标准越具体，复盘越有价值。

## 5. 证据矩阵

| 问题 | 需要的证据 |
| --- | --- |
| 是否发生 | 原始日志、事件 ID |
| 谁执行 | actor、session、设备 |
| 做了什么 | action、参数、资源 |
| 影响多大 | 数据范围、权限范围 |
| 是否被发现 | 告警、规则、时间 |
| 是否被处置 | 工单、操作记录、结果 |

紫队活动应保留完整证据链，方便后续转成检测规则和 runbook。

## 6. 检测评估

评估检测不要只看是否告警。

| 维度 | 问题 |
| --- | --- |
| Coverage | 是否有日志覆盖 |
| Fidelity | 告警是否准确 |
| Context | 是否提供调查上下文 |
| Timeliness | 是否足够及时 |
| Actionability | 是否能指导响应动作 |
| Resilience | 攻击者轻微变化后是否仍有效 |

一个只写着“可疑行为”的告警，通常不能算高质量检测。

## 7. 安全边界

紫队演练必须控制风险：

- 明确禁止破坏性动作。
- 使用测试账号和测试资源。
- 避免影响生产可用性。
- 约定演练窗口。
- 预设停止条件。
- 保留回滚方案。

真实度和安全性需要平衡。目标是验证控制，不是制造事故。

## 8. 改进闭环

每个发现都要落到一类工程动作：

| 缺口 | 动作 |
| --- | --- |
| 没有日志 | 增加数据源或字段 |
| 有日志无告警 | 新增检测规则 |
| 告警无上下文 | 丰富资产、身份、owner 信息 |
| 响应慢 | 更新 runbook 和权限 |
| 控制绕过 | 修复策略或加补偿控制 |
| 权限过大 | 降权或 JIT |

紫队的价值体现在改进关闭率，而不是活动次数。

## 9. 指标

| 指标 | 含义 |
| --- | --- |
| Scenario pass rate | 场景验证通过率 |
| Detection latency | 检测延迟 |
| Triage accuracy | 分析判断准确性 |
| Control gap closure | 缺口关闭率 |
| Retest success rate | 修复后复测通过率 |
| ATT&CK coverage quality | 技术覆盖质量 |

不要只统计 ATT&CK 覆盖数量。覆盖质量比覆盖格子更重要。

## 10. 节奏

建议每月做小场景，每季度做跨团队场景，每半年做综合演练。

好的紫队机制会让攻击知识变成防御能力，让一次演练变成长期资产。
