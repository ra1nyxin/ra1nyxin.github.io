# Compliance Evidence Engineering

合规证据工程的目标是让控制措施可证明，审计前临时截图风险很高。成熟的合规体系应该尽量自动收集证据、保持证据可追溯，并把合规要求转化为日常安全控制。

## 1. 证据类型

| 证据 | 示例 |
| --- | --- |
| Configuration | MFA 策略、日志保留配置 |
| Process | 变更审批、权限复核记录 |
| Runtime | 告警处理、备份任务结果 |
| Access | 用户权限、管理员操作 |
| Training | 培训完成和测试记录 |
| Exception | 风险例外和补偿控制 |

证据必须能证明控制在审计周期内持续有效。

## 2. 自动化优先

适合自动采集：

- 云配置。
- 身份策略。
- 设备合规状态。
- 漏洞修复状态。
- 日志保留设置。
- 备份任务状态。
- 权限复核结果。

人工截图容易过期、难以复现，也难以证明连续性。

## 3. 证据字段

每份证据应记录：

```text
control_id
evidence_type
source_system
collection_time
scope
owner
result
link_or_artifact
retention_period
```

证据没有范围说明时，很难证明覆盖了哪些系统。

## 4. 控制映射

一个控制可以映射多个框架：

- ISO 27001。
- SOC 2。
- PCI DSS。
- HIPAA。
- GDPR。
- 内部安全基线。

避免为每个框架重复建设证据。核心是控制本身有效。

## 5. 例外管理

合规例外需要：

- 风险描述。
- 业务原因。
- 补偿控制。
- owner。
- 过期时间。
- 审批记录。
- 复核记录。

没有过期时间的例外会变成永久绕过。

## 6. 审计准备

审计前检查：

- 证据是否覆盖完整范围。
- 证据时间是否在审计周期内。
- owner 是否能解释控制。
- 例外是否仍然有效。
- 证据是否包含敏感信息。
- 链接是否可访问。

好的证据工程能让审计准备变成验证，减少临时补材料。

## 7. 与安全运营联动

合规证据应来自真实运营：

- 权限复核来自身份治理。
- 漏洞证据来自漏洞平台。
- 日志证据来自日志架构。
- 备份证据来自恢复演练。
- 事件证据来自 IR 流程。

如果合规证据和日常安全系统分离，长期维护成本会很高。

## 8. 指标

| 指标 | 目标 |
| --- | --- |
| 自动证据比例 | 持续提升 |
| 证据缺口数量 | 持续下降 |
| 例外超期数量 | 0 |
| 控制 owner 覆盖 | 100% |
| 证据复用率 | 持续提升 |
| 审计发现重复率 | 持续下降 |

合规证据工程的目标是让安全控制自然产生证据，避免为了审计临时制造证据。
