# Business Continuity and Security

业务连续性和安全不是两个独立领域。安全事件会影响可用性，恢复过程也会引入新的安全风险。成熟组织需要把安全事件、灾难恢复、备份、身份恢复和沟通机制统一设计。

## 1. 关键问题

先回答：

- 哪些业务必须优先恢复。
- 每个业务的 RTO 和 RPO 是多少。
- 依赖哪些身份、网络、云和数据系统。
- 恢复时需要哪些权限。
- 最小可运行版本是什么。
- 对客户和监管的通知要求是什么。

没有业务优先级，恢复现场会变成资源争夺。

## 2. 依赖图

关键业务要维护依赖图：

| 依赖 | 示例 |
| --- | --- |
| Identity | SSO、MFA、目录服务 |
| Network | DNS、VPN、网关 |
| Data | 数据库、对象存储、备份 |
| Compute | 云账号、集群、虚机 |
| CI/CD | 发布系统、制品库 |
| Observability | 日志、监控、告警 |

恢复计划必须包含这些基础依赖，而不仅是业务应用本身。

## 3. 安全事件下的恢复

恢复前确认：

- 攻击路径是否关闭。
- 受影响凭证是否轮换。
- 持久化是否清理。
- 备份是否可信。
- 恢复环境是否隔离。
- 监控是否增强。

在未确认根因前恢复，可能导致二次失陷。

## 4. 最小运行模式

为关键业务定义：

- 最小功能集合。
- 只读模式。
- 手工审批流程。
- 降级依赖。
- 限制高风险操作。
- 临时访问控制。

最小运行模式能在完整恢复前保持核心服务。

## 5. 通信计划

需要明确：

- 内部沟通渠道。
- 客户通知模板。
- 监管通知条件。
- 供应商联系人。
- 管理层汇报节奏。
- 单一事实来源。

事件中沟通混乱会放大业务影响。

## 6. 演练

演练主题：

- 身份系统不可用。
- 云账号被锁定。
- 核心数据库损坏。
- 勒索导致文件服务不可用。
- 日志平台不可用。
- 关键供应商中断。

演练要记录恢复时间、阻塞点和需要改进的权限或流程。

## 7. 安全控制不中断

恢复期间不能完全关闭安全控制。

至少保留：

- 强认证。
- 管理操作审计。
- 关键日志。
- 变更记录。
- 临时权限过期。
- 高风险操作审批。

紧急状态下创建的临时权限，事件后必须清理。

## 8. 指标

| 指标 | 目标 |
| --- | --- |
| 关键业务 RTO/RPO 覆盖 | 100% |
| 恢复演练通过率 | 持续提升 |
| 依赖图更新频率 | 定期 |
| 临时权限清理率 | 100% |
| 最小运行模式覆盖 | 关键业务优先 |
| 复盘改进关闭率 | 100% |

业务连续性的核心是让组织在异常情况下仍然能安全地运行和恢复。
