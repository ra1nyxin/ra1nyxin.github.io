# Nmap 使用笔记

Nmap 适合在授权范围内做资产发现、端口确认、服务识别和基础脚本检查。它的结果反映的是“从当前网络位置看到的状态”，不是资产的绝对真相。云安全组、防火墙、负载均衡、ACL、端口敲门和网络分区都会影响结果，所以重要目标最好从不同位置交叉验证。

第一步通常是主机发现。`-sn` 不做端口扫描，只判断哪些主机在线。内网梳理网段时很快。

```bash
nmap -sn 192.168.1.0/24
```

如果 ICMP、ARP 或探测包被拦，Nmap 可能误判主机不在线。明确知道目标存在时，用 `-Pn` 跳过主机发现，直接扫描端口。

```bash
nmap -Pn 192.168.1.10
```

默认扫描会看一批常见 TCP 端口，适合作为第一眼画像。要确认完整暴露面，还是要扫全端口。

```bash
nmap 192.168.1.10
nmap -p 22,80,443,8080 192.168.1.10
nmap -p 1-10000 192.168.1.10
nmap -p- 192.168.1.10
```

我常用的节奏是先扫全端口，只拿端口列表；再针对开放端口做服务识别和默认脚本。这样比一开始就 `-sV -sC -p-` 更可控。

```bash
nmap -p- --min-rate 3000 -oN ports.txt 192.168.1.10
nmap -sV -sC -p 22,80,443 -oN services.txt 192.168.1.10
```

服务识别用 `-sV`，它比端口号可靠。比如 8443 不一定是 HTTPS，80 也可能不是普通 Web。

```bash
nmap -sV -p 22,80,443,8443 192.168.1.10
nmap -sV --version-intensity 8 -p 8080 192.168.1.10
```

默认脚本 `-sC` 会跑一组相对常规的 NSE 脚本。它适合基础信息收集，但不是“安全扫描全家桶”。生产目标上跑脚本前要知道脚本会做什么。

```bash
nmap -sC -sV -p 22,80,443 192.168.1.10
```

HTTP 场景里，`http-title`、`http-server-header` 这类脚本很实用，可以快速区分站点内容。

```bash
nmap --script http-title,http-server-header -p 80,443,8080 192.168.1.10
```

查脚本说明用 `--script-help`，不要看到脚本名像安全检查就直接跑。

```bash
nmap --script-help http-title
```

UDP 扫描慢，而且结果更容易受丢包影响。先扫常见端口，再按业务线索扩大范围。

```bash
nmap -sU -p 53,67,68,69,123,161,500,4500 192.168.1.10
nmap -sU --top-ports 100 192.168.1.10
```

速率参数要按网络质量调。内网稳定目标可以适当提高 `--min-rate`、降低重试；公网或跨地域链路太激进会漏结果。

```bash
nmap -p- --min-rate 3000 --max-retries 2 192.168.1.10
nmap -T3 -p- 192.168.1.10
```

输出建议用 `-oA` 同时保存普通文本、grepable 和 XML。后续写报告看文本，工具处理用 XML。

```bash
nmap -sV -oA scan-result 192.168.1.10
nmap -sV -oN scan-result.txt 192.168.1.10
nmap -sV -oX scan-result.xml 192.168.1.10
```

扫描多个目标时，把范围放进文件更清楚，也方便留存。

```bash
nmap -iL targets.txt -p 80,443 -oA web-scan
```

排查结果时要看端口状态含义。`open` 是有服务响应，`closed` 是主机可达但端口关闭，`filtered` 通常是包被过滤或没有得到明确响应，`open|filtered` 常见于 UDP。不要把 filtered 直接当成端口开放。

我一般按这个流程走：主机发现，TCP 全端口，服务识别，按服务补 NSE，必要时 UDP，最后保存结构化输出并和资产清单对照。Nmap 扫描只是入口，真正结论还要结合配置、云安全组、应用日志和实际业务归属。
