# Zero Trust Architecture

零信任不是采购某个产品，也不是把所有系统前面都套一层代理。它是一套架构原则：默认不信任网络位置，持续验证身份、设备、权限、上下文和资源风险。

零信任落地时最容易失败的原因，是把它做成一次性改造项目。更合理的方式是把它拆成可度量的控制面，逐步替换隐式信任。

## 1. 核心原则

| 原则 | 含义 |
| --- | --- |
| Explicit verification | 每次访问都基于身份、设备、位置、风险重新判断 |
| Least privilege | 权限按任务和时间收敛 |
| Assume breach | 默认内部网络也可能被攻陷 |
| Continuous monitoring | 决策依赖实时遥测 |
| Strong segmentation | 资源之间不能默认互通 |

零信任的关键不是“没有信任”，而是“信任必须有证据，而且会过期”。

## 2. 架构组件

| 组件 | 职责 |
| --- | --- |
| Identity provider | 统一身份、认证和条件访问 |
| Device posture | 判断设备是否托管、加密、合规 |
| Policy engine | 根据上下文做访问决策 |
| Policy enforcement point | 执行允许、拒绝、降权、重新认证 |
| Resource gateway | 保护应用、API、数据库、管理面 |
| Telemetry platform | 收集访问、设备、身份和行为数据 |

这些组件可以来自不同系统，但策略语义必须一致。否则用户看到的是多个互相矛盾的安全门。

## 3. 访问决策模型

一次访问至少应考虑：

```text
subject: 用户、服务、工作负载
device: 设备合规状态
resource: 被访问资源敏感度
action: 读取、写入、管理、导出
context: 位置、时间、网络、风险评分
session: 当前会话强度和持续时间
```

示例策略：

| 场景 | 决策 |
| --- | --- |
| 托管设备访问普通内部系统 | 允许 |
| 非托管设备访问敏感数据 | 只读或拒绝 |
| 管理员从新地区登录 | 重新认证并告警 |
| 服务账号访问非预期资源 | 拒绝并触发调查 |
| 高风险用户导出大量数据 | 阻断或要求审批 |

## 4. 分段策略

传统网络分段通常按网段划分，零信任更关注资源关系。

建议从这些边界开始：

- 用户到管理面。
- CI/CD 到生产环境。
- 应用到数据库。
- SaaS 到企业数据。
- 第三方到内部资源。
- 工作负载到云控制面。

分段目标不是制造复杂网络，而是让横向移动需要经过明确授权点。

## 5. 迁移路径

推荐分阶段推进：

1. 统一身份源，清理孤儿账号。
2. 管理员和关键系统启用强 MFA。
3. 建立设备合规判断。
4. 把高价值内部应用接入访问网关。
5. 对生产管理面实施 Just-in-Time 权限。
6. 对服务到服务访问引入 workload identity。
7. 用日志验证策略是否有效。

不要试图一口气覆盖全部系统。优先选择高价值、边界清晰、用户群可控的应用。

## 6. 常见误区

| 误区 | 后果 |
| --- | --- |
| 只买 ZTNA 产品 | 解决不了权限和数据问题 |
| 只看用户访问 | 忽略服务账号和自动化链路 |
| 过度依赖 IP allowlist | 内网失陷后仍然高风险 |
| 策略过于粗糙 | 用户绕过或安全例外泛滥 |
| 缺少日志闭环 | 无法证明策略有效 |

零信任如果没有可观测性，就很难从“配置了策略”走向“风险真的降低”。

## 7. 度量指标

| 指标 | 解释 |
| --- | --- |
| 高价值应用纳管比例 | 访问是否经过统一策略 |
| 管理面 JIT 覆盖率 | 管理权限是否按需发放 |
| 非托管设备访问阻断率 | 设备信任是否生效 |
| 服务身份长期密钥数量 | 工作负载身份是否现代化 |
| 横向访问路径数量 | 分段是否收敛 |
| 策略例外数量 | 设计是否可持续 |

成熟的零信任体系应该让高风险访问更难，让正常访问更清晰，而不是让所有访问都变慢。
