# Database Security Baseline

数据库安全的重点是身份、网络、权限、加密、审计、备份和变更控制。数据库往往承载最高价值数据，任何绕过应用层的访问都需要严格治理。

## 1. 访问边界

数据库不应直接暴露公网。访问应通过：

- 应用服务。
- 受控管理通道。
- 堡垒或零信任网关。
- 临时授权机制。
- 审计代理或数据库审计。

开发者直连生产数据库应作为例外，日常访问走受控通道。

## 2. 账号模型

| 账号 | 要求 |
| --- | --- |
| Application | 最小权限，只访问所需 schema |
| Admin | 强认证，JIT，完整审计 |
| Read-only | 限制导出和敏感表 |
| Migration | 只在发布窗口使用 |
| Break-glass | 极少使用，登录即告警 |

共享数据库账号会严重削弱追责能力。

## 3. 权限控制

原则：

- 应用账号不使用 DBA 权限。
- 读取、写入、DDL、导出分离。
- 敏感表单独授权。
- 定期清理未使用账号。
- 权限变更进入审计。

权限设计要按业务动作建模，不要只按数据库角色粗放授权。

## 4. 加密

要求：

- 传输加密。
- 存储加密。
- 备份加密。
- 密钥权限和数据库权限分离。
- 高敏字段可考虑字段级加密或 tokenization。

加密不能替代访问控制，但能降低存储层泄露影响。

## 5. 审计日志

优先记录：

- 登录成功和失败。
- 管理操作。
- 权限变更。
- DDL 变更。
- 敏感表查询。
- 大批量导出。
- 备份和恢复操作。

审计日志必须防篡改，并能关联用户、应用和请求。

## 6. 变更管理

数据库变更要求：

- migration 脚本评审。
- 高风险 DDL 有回滚方案。
- 生产变更窗口。
- 数据修复操作记录原因。
- 大规模更新先 dry-run。

数据库事故很多来自正常权限下的错误操作。

## 7. 备份和恢复

要求：

- 定期备份。
- 备份加密。
- 跨区域或跨账号保存。
- 定期恢复演练。
- 备份删除受控。
- 备份访问有审计。

不能恢复的备份对业务连续性没有价值。

## 8. 检测场景

优先检测：

- 非应用来源访问生产数据库。
- 管理员在非维护窗口执行 DDL。
- 普通账号读取大量敏感记录。
- 新账号获得高权限。
- 数据库快照被复制到低安全环境。
- 备份被删除或导出。

数据库检测要结合数据分类和访问上下文。

## 9. 指标

| 指标 | 目标 |
| --- | --- |
| 公网暴露数据库 | 0 |
| 共享账号数量 | 持续下降 |
| 管理操作审计覆盖 | 100% |
| 敏感表访问审计 | 100% |
| 恢复演练通过率 | 持续提升 |
| 未使用高权限账号 | 0 |

数据库安全的核心是让高价值数据访问可控、可查、可恢复。
