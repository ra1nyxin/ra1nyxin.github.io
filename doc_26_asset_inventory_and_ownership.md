# Asset Inventory and Ownership

资产清单是安全工程的地基。没有准确资产清单，漏洞治理、日志覆盖、权限复核、事件响应和成本管理都会变成局部猜测。

资产管理的目标不是维护一张静态表，而是让每个系统、账号、数据集和暴露面都有 owner、用途、风险等级和生命周期。

## 1. 资产类型

| 类型 | 示例 | 关键字段 |
| --- | --- | --- |
| Internet-facing | 域名、IP、CDN、API 网关 | owner、业务、证书、暴露端口 |
| Cloud resource | VM、Bucket、数据库、KMS key | 账号、区域、标签、权限 |
| Application | Web、后台、内部工具 | owner、数据等级、依赖 |
| Identity | 用户、服务账号、机器人账号 | owner、权限、最近使用 |
| Data | 数据库、对象存储、数据集 | 分类、保留期、共享方 |
| Endpoint | 笔记本、服务器、移动设备 | 使用人、状态、补丁 |

不同资产可以来自不同系统，但最终要能关联到统一 owner。

## 2. Owner 模型

每个资产至少需要两个 owner：

- Business owner：决定业务价值、可用性要求和数据责任。
- Technical owner：负责配置、修复、监控和变更。

只有团队名没有具体责任人时，事件响应中很容易没人接手。

## 3. 标签规范

云和内部平台建议统一标签：

```text
owner
service
environment
data_classification
criticality
cost_center
created_by
expires_at
```

标签不应该只用于计费，也应该服务于安全决策。

## 4. 发现方式

资产发现可以组合：

- 云账号 API 枚举。
- DNS 和证书透明日志。
- 外部攻击面扫描。
- 代码仓库和 CI/CD 元数据。
- 身份平台和 SaaS 管理接口。
- EDR 和 MDM 设备清单。
- 数据平台目录。

单一来源通常不完整，需要交叉验证。

## 5. 生命周期

| 阶段 | 控制 |
| --- | --- |
| 创建 | 必须指定 owner 和环境 |
| 变更 | 关键字段变更进入审计 |
| 运行 | 定期校验状态和暴露面 |
| 闲置 | 长期无流量或无使用进入清理 |
| 下线 | 回收权限、数据、DNS、证书和监控 |

下线流程要特别关注遗留 DNS、token、存储桶和监控静默。

## 6. 风险分级

资产风险可以按这些因素判断：

- 是否互联网暴露。
- 是否承载敏感数据。
- 是否拥有高权限。
- 是否业务关键。
- 是否缺少日志。
- 是否缺少 owner。
- 是否长期未维护。

高风险资产应优先纳入漏洞扫描、日志采集和响应演练。

## 7. 常见问题

| 问题 | 影响 |
| --- | --- |
| 没有 owner | 漏洞和事件无人处理 |
| 标签随意 | 无法自动化治理 |
| 临时资源不下线 | 暴露面扩大 |
| 影子 SaaS | 数据和账号脱离管控 |
| 资产系统不同步 | 决策基于过期信息 |

资产清单的质量决定很多安全流程的上限。

## 8. 指标

| 指标 | 目标 |
| --- | --- |
| owner 覆盖率 | 100% |
| 关键标签完整率 | 持续提升 |
| 未登记互联网资产 | 0 |
| 长期闲置资产 | 持续下降 |
| 高风险资产日志覆盖 | 100% |
| 下线流程完成率 | 100% |

资产管理不是 CMDB 项目，而是安全控制能否准确落地的基础能力。
