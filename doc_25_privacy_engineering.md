# Privacy Engineering

隐私工程要把数据最小化、目的限制、用户权利、保留期限和可审计性落到系统设计中，不能只停留在合规文档。隐私做得好，数据泄露的影响范围也会自然下降。

## 1. 基本原则

| 原则 | 含义 |
| --- | --- |
| Data minimization | 只收集必要数据 |
| Purpose limitation | 数据只用于明确目的 |
| Storage limitation | 到期删除或匿名化 |
| Transparency | 用户知道数据如何被使用 |
| Access control | 只有需要的人和系统能访问 |
| Accountability | 数据处理可审计 |

隐私原则要转化为字段、接口、日志和流程。

## 2. 数据映射

需要知道：

- 收集哪些个人数据。
- 数据来自哪里。
- 存在哪里。
- 谁能访问。
- 用于什么目的。
- 保留多久。
- 是否共享给第三方。
- 删除请求如何传播。

没有数据地图，隐私请求和事件响应都会很慢。

## 3. 最小化设计

常见做法：

- 不收集不需要字段。
- 能本地计算就不上传。
- 能用统计值就不保存明细。
- 能用短期 token 就不用长期标识符。
- 测试环境使用脱敏数据。
- 日志避免记录完整个人信息。

最小化的重点是减少无意义的数据负债，同时保留必要业务价值。

## 4. 用户权利

系统应支持：

- 查询个人数据。
- 更正错误数据。
- 删除或注销。
- 导出数据。
- 撤回同意。
- 限制处理。

这些能力如果完全靠人工数据库操作，会带来一致性和审计风险。

## 5. 保留与删除

删除设计要考虑：

- 主库。
- 缓存。
- 搜索索引。
- 数据仓库。
- 日志。
- 备份。
- 第三方系统。

删除不一定意味着立即从所有备份物理清除，但策略要明确、可解释、可执行。

## 6. 脱敏与访问

不同角色看到不同数据：

| 角色 | 建议 |
| --- | --- |
| 客服 | 只显示处理所需字段 |
| 分析师 | 使用聚合或脱敏数据 |
| 工程师 | 默认无生产个人数据访问 |
| 管理员 | 高风险访问审计 |
| 第三方 | 最小字段和合同约束 |

生产个人数据访问要有理由、审批和审计。

## 7. 日志隐私

日志常见问题：

- URL 参数包含个人信息。
- 错误日志打印请求体。
- 分析事件记录完整标识符。
- 第三方 SDK 收集过多字段。
- 调试日志长期保留。

日志设计要同时满足调查和最小化，必要时使用 tokenization 或字段级脱敏。

## 8. Privacy by Design

设计评审中加入隐私问题：

- 这个字段是否必须收集。
- 用户是否知道使用目的。
- 是否有更低敏感度替代方案。
- 谁能访问原始数据。
- 删除请求如何影响下游。
- 数据泄露后影响范围多大。

隐私评审越早做，架构代价越低。

## 9. 指标

| 指标 | 目标 |
| --- | --- |
| 数据地图覆盖率 | 高风险系统 100% |
| 未分类个人数据集 | 0 |
| 删除请求 SLA | 可证明 |
| 生产数据访问审计率 | 100% |
| 测试环境真实个人数据 | 接近 0 |
| 第三方共享清单 | 完整 |

隐私工程要让数据使用清楚、克制、可证明，避免事故后临时解释。
