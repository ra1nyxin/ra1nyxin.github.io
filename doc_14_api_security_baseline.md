# API Security Baseline

API 是现代系统的业务入口，也是权限、数据和自动化能力的集中暴露面。API 安全基线的目标不是阻止所有复杂攻击，而是让常见高影响风险在设计、开发、上线和运行阶段都能被系统性控制。

## 1. API 资产管理

先知道有哪些 API。

每个 API 应记录：

- owner。
- 用途。
- 认证方式。
- 权限模型。
- 数据分类。
- 暴露范围。
- 调用方。
- 版本。
- 下线计划。

未登记 API 往往缺少日志、限流和权限复核，是最容易成为盲点的资产。

## 2. 认证与授权

认证回答“你是谁”，授权回答“你能做什么”。

| 控制 | 要求 |
| --- | --- |
| Authentication | 禁止匿名访问敏感 API |
| Authorization | 服务端做对象级和动作级检查 |
| Token scope | token 权限最小化 |
| Token lifetime | access token 短期有效 |
| Service identity | 服务间调用使用明确身份 |
| Admin action | 高风险动作重新认证或审批 |

不要只在前端隐藏按钮。所有授权判断必须在服务端执行。

## 3. 对象级授权

API 最常见的问题之一是对象级越权。

检查问题：

- 用户能否访问不属于自己的对象 ID。
- 管理员接口是否只校验登录而不校验角色。
- 多租户数据是否按租户隔离。
- 批量接口是否逐项检查权限。
- 导出接口是否复用普通读取权限。

对象级授权应尽量封装在统一数据访问层，避免每个接口手写一遍。

## 4. 输入校验

输入校验应基于 schema。

要求：

- 明确字段类型、长度、范围。
- 拒绝未知字段或明确处理。
- 对文件上传限制大小、类型和解析资源。
- 对 URL、路径、命令参数做 allowlist。
- 对 JSON 嵌套深度做限制。
- 错误信息不泄露内部细节。

输入校验不是为了让错误提示更好看，而是为了收紧可被利用的解释空间。

## 5. 限流与滥用防护

限流要按身份、IP、租户和资源组合设计。

| 场景 | 控制 |
| --- | --- |
| 登录接口 | 账号和来源双维度限速 |
| 搜索接口 | 查询复杂度限制 |
| 导出接口 | 审批、队列、频率限制 |
| 写入接口 | 幂等键、配额 |
| 高成本计算 | 任务队列和预算 |

单纯按 IP 限流对企业代理、移动网络和自动化攻击都不够。

## 6. 错误处理

错误响应应一致：

- 不暴露堆栈。
- 不泄露 SQL、路径、内部服务名。
- 不区分账号是否存在，除非业务必须。
- 返回稳定错误码。
- 为服务端日志保留 request_id。

用户看到的是简洁错误，调查人员看到的是完整上下文。

## 7. 日志与审计

高价值 API 至少记录：

| 字段 | 说明 |
| --- | --- |
| request_id | 请求追踪 |
| actor | 调用者 |
| client_id | 应用或服务 |
| action | 行为 |
| resource | 目标资源 |
| outcome | 成功或失败 |
| source_ip | 来源 |
| user_agent | 客户端 |
| latency | 性能和异常分析 |

敏感字段不应原样写入日志，尤其是 token、密码、身份证件、支付信息。

## 8. 版本和废弃

API 版本治理要求：

- 新版本不默认继承旧版本风险。
- 旧版本有明确下线时间。
- 废弃版本仍然保持安全补丁。
- 外部调用方有迁移通知。
- 文档和实际行为一致。

废弃 API 如果还在运行，就仍然是生产攻击面。

## 9. 测试清单

上线前至少验证：

- 未登录访问是否被拒绝。
- 低权限用户是否不能访问高权限动作。
- 跨租户对象 ID 是否被拒绝。
- 批量接口是否逐项鉴权。
- 大输入是否被限制。
- token 过期和撤销是否生效。
- 日志是否记录足够调查字段。
- 错误响应是否不泄露内部信息。

这些测试适合自动化加入 CI。

## 10. 运行期检测

优先检测：

- 单账号访问大量对象。
- API token 调用范围突然扩大。
- 低频接口突然高频调用。
- 连续授权失败后成功。
- 非预期地区调用管理 API。
- 导出接口异常使用。

API 安全基线的核心是把权限和数据控制做成默认能力，而不是依赖每个接口开发者临时记得。
