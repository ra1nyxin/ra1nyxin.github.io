# Web Application Security Review

Web 安全评审不应该只跑扫描器。扫描器能发现一部分显性问题，但业务授权、数据流、状态机、异步任务和第三方集成需要人工理解。

## 1. 评审范围

先确认：

- 应用入口。
- 用户角色。
- 数据类型。
- 管理接口。
- 第三方集成。
- 文件上传和下载。
- 导入导出流程。
- 支付、优惠、积分等业务状态。

范围不清晰时，评审容易只覆盖最明显页面。

## 2. 身份与会话

检查点：

- 登录失败是否限速。
- 密码重置是否可被枚举或绕过。
- MFA 是否覆盖高风险动作。
- session 是否合理过期。
- logout 后 token 是否失效。
- cookie 是否设置 Secure、HttpOnly、SameSite。

会话安全要结合浏览器行为和服务端状态一起看。

## 3. 授权

重点检查：

| 类型 | 问题 |
| --- | --- |
| Horizontal | 用户访问其他用户资源 |
| Vertical | 普通用户执行管理员操作 |
| Tenant | 跨租户访问 |
| Object | 只改 ID 就能访问 |
| Function | 隐藏按钮但接口仍可调用 |

授权漏洞通常比输入型漏洞更贴近业务影响。

## 4. 输入与输出

检查：

- 所有输入是否有 schema。
- 富文本是否有安全清洗。
- 文件类型是否按内容验证。
- 下载文件名是否安全处理。
- 输出是否正确编码。
- 错误页是否泄露堆栈。

输入校验和输出编码解决的是不同问题，不能互相替代。

## 5. 文件处理

高风险点：

- 上传可执行文件。
- 文件解析器资源消耗。
- 压缩包路径穿越。
- 图片元数据泄露。
- 下载接口越权。
- 临时文件未清理。

文件上传最好进入隔离存储，后续处理由低权限任务完成。

## 6. 业务逻辑

业务安全问题需要理解流程：

- 优惠券是否可重复使用。
- 订单状态能否跳转。
- 审批流程能否绕过。
- 退款金额是否可篡改。
- 邀请和权限是否可滥用。
- 批量操作是否逐项鉴权。

业务逻辑测试不能只依赖通用漏洞字典。

## 7. API 与前端

前端代码会暴露很多线索：

- 未公开 API 路径。
- feature flag。
- 第三方 key。
- 角色判断逻辑。
- 调试接口。

但前端逻辑不能作为安全边界。所有关键判断必须在服务端重新执行。

## 8. 安全响应头

常见响应头：

- Content-Security-Policy。
- Strict-Transport-Security。
- X-Content-Type-Options。
- Referrer-Policy。
- Permissions-Policy。

响应头不能修复业务漏洞，但能降低浏览器侧攻击面。

## 9. 评审输出

输出建议包含：

- 风险标题。
- 影响范围。
- 前提条件。
- 复现摘要。
- 根因。
- 修复建议。
- 验证方式。
- 优先级。

修复建议要指向代码或设计，“加强校验”这种写法不够。

## 10. 常见高价值测试

- 普通用户改 URL ID 访问其他资源。
- 批量接口混入无权限对象。
- 管理 API 直接调用。
- 导出接口绕过页面限制。
- 重放旧状态请求。
- 用低权限 token 调用高权限动作。

Web 评审的价值在于把技术漏洞和业务影响连起来。
