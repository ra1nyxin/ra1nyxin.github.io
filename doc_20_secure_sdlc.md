# Secure SDLC

安全开发生命周期不是把安全扫描塞进 CI 就完成了。它要把安全要求、设计评审、依赖管理、代码审查、测试、发布门禁和运行期反馈连接起来。

## 1. 生命周期控制点

| 阶段 | 控制 |
| --- | --- |
| Requirement | 定义安全需求和数据等级 |
| Design | 威胁建模和权限设计 |
| Development | 安全编码和依赖控制 |
| Review | 代码审查和安全评审 |
| Testing | SAST、DAST、IAST、单元安全测试 |
| Release | 发布门禁和制品签名 |
| Runtime | 监控、漏洞治理和反馈 |

安全越早进入流程，修复成本越低。

## 2. 安全需求

安全需求要写成可验证项：

- 所有管理接口必须鉴权。
- 敏感导出需要审计。
- API token 需要 scope 和过期时间。
- 用户输入必须 schema 校验。
- 高风险操作需要重新认证。
- 关键日志保留不少于指定周期。

模糊要求会在开发后期变成争论。

## 3. 代码审查

安全审查重点：

- 鉴权是否在服务端执行。
- 对象级授权是否完整。
- 输入校验是否覆盖边界。
- 错误信息是否泄露内部细节。
- secret 是否进入代码或日志。
- 高风险依赖是否新增。
- 日志是否足够调查。

代码审查不应该只关注语法和风格。

## 4. 自动化扫描

| 工具类型 | 价值 | 局限 |
| --- | --- | --- |
| SAST | 找代码模式问题 | 误报和上下文不足 |
| SCA | 找依赖漏洞 | 需要判断可利用性 |
| Secret scan | 找泄露凭证 | 需要快速轮换 |
| IaC scan | 找配置风险 | 需要结合环境 |
| DAST | 找运行时问题 | 覆盖依赖测试路径 |

扫描结果要进入治理流程，否则只是噪声。

## 5. 发布门禁

门禁策略：

- Critical 漏洞默认阻断。
- secret 命中默认阻断。
- 未签名产物不能进入生产。
- 高风险权限变更需要审批。
- 安全例外必须有过期时间。

门禁要可解释。阻断规则如果经常误伤，团队会绕过它。

## 6. 依赖管理

要求：

- 锁定依赖版本。
- 使用可信 registry。
- 监控新增依赖。
- 定期更新基础框架。
- 识别未维护组件。
- 记录依赖 owner。

依赖治理不是追逐最新版本，而是降低未知和不可维护风险。

## 7. 安全测试用例

应把安全场景写成测试：

- 普通用户不能访问其他租户对象。
- token 过期后请求失败。
- 批量接口逐项鉴权。
- 输入超长或异常类型被拒绝。
- 导出操作产生审计日志。
- 权限变更立即生效。

安全测试越贴近业务逻辑，价值越高。

## 8. 运行期反馈

运行期发现的问题应回流：

- 告警误报回到检测规则。
- 漏洞复发回到编码规范。
- 事件根因回到设计模板。
- 修复困难回到架构债务。
- 新攻击路径回到测试用例。

SSDLC 是闭环，不是线性流程。

## 9. 指标

| 指标 | 目标 |
| --- | --- |
| 安全需求覆盖率 | 高风险项目 100% |
| 威胁建模覆盖率 | 关键系统 100% |
| secret 阻断率 | 100% |
| 高危漏洞超期率 | 持续下降 |
| 安全测试通过率 | 稳定提升 |
| 例外过期数量 | 0 |

成熟的 SSDLC 会让安全成为工程默认路径，而不是发布前的临时检查。
