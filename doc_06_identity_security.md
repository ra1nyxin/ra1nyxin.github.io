# Identity Security Engineering

身份安全是现代安全体系的主干。云、SaaS、CI/CD、远程办公和自动化平台把传统边界拆散以后，真正稳定的控制点通常变成了账号、权限、会话、设备状态和授权链路。

身份安全不能只开 MFA，还要持续回答四个问题：

1. 谁正在访问？
2. 访问是否符合当前上下文？
3. 权限是否刚好够用？
4. 异常访问能否快速发现和收敛？

## 1. 资产分层

身份资产需要先分层，否则权限治理会变成账号清单维护。

| 类型 | 例子 | 主要风险 |
| --- | --- | --- |
| Human identity | 员工、外包、管理员 | 凭证泄露、权限膨胀、离职遗留 |
| Service identity | 服务账号、API token、机器人账号 | 长期密钥、不可追责、横向移动 |
| Workload identity | 容器、函数、虚机、CI job | 运行时越权、临时凭证滥用 |
| Privileged identity | Root、Owner、Global Admin | 单点失陷导致全局影响 |
| Break-glass identity | 应急账号 | 缺少监控、平时未验证可用性 |

建议先把身份资产做成可查询台账，至少包含 owner、用途、权限范围、凭证类型、最近使用时间、轮换周期和告警策略。

## 2. 权限设计原则

最小权限不应该停留在口号层面，而应该落实到权限生命周期。

| 阶段 | 控制点 |
| --- | --- |
| 申请 | 说明业务理由、资源范围、过期时间 |
| 审批 | 由资源 owner 或数据 owner 审批 |
| 授权 | 默认使用角色或组，不直接授予个人 |
| 使用 | 高风险操作要求强认证和审计 |
| 复核 | 定期确认仍然需要 |
| 回收 | 到期自动回收，离职立即回收 |

权限要尽量具备过期时间。永久权限越多，真实攻击面越难解释。

## 3. MFA 与强认证

MFA 的价值取决于抗钓鱼能力和覆盖面。

| 方式 | 安全性 | 建议 |
| --- | --- | --- |
| SMS OTP | 低 | 只作为过渡方案 |
| App OTP | 中 | 可用于普通账号 |
| Push MFA | 中 | 需要防疲劳攻击策略 |
| FIDO2 / Passkey | 高 | 优先用于管理员和高风险系统 |
| 设备证书 | 高 | 适合企业托管设备 |

关键要求：

- 管理员账号必须使用抗钓鱼 MFA。
- MFA 重置流程必须纳入审计。
- 不要让备用邮箱或客服流程绕过强认证。
- 对异常地理位置、异常设备、异常 ASN 做条件访问。

## 4. 服务账号治理

服务账号常常比人类账号更危险，因为它们长期存在、权限高、行为模式稳定，容易被忽略。

服务账号应满足：

- 有明确 owner。
- 有用途说明。
- 权限范围可解释。
- 凭证可轮换。
- 最近使用时间可见。
- 不允许交互式登录，除非有明确理由。
- 不允许多人共享同一账号进行人工操作。

如果一个服务账号无法说明用途，先降权观察，保留前必须补齐用途说明。

## 5. 会话安全

身份失陷不只来自密码泄露，也可能来自 token、cookie、refresh token 或 OAuth grant。

需要关注：

- 高风险操作前重新认证。
- Refresh token 有合理有效期。
- 设备丢失后能撤销会话。
- OAuth 应用授权有审批和复核。
- 异常 token 使用可以被检测。

典型检测：

```text
同一账号在短时间内从两个远距离地区使用 refresh token
管理员账号创建 OAuth 应用后立即访问大量数据
低频服务账号突然在非工作时间调用高权限 API
```

## 6. 身份日志

身份日志是检测工程中优先级最高的数据源之一。

| 日志 | 用途 |
| --- | --- |
| Login events | 发现异常登录和爆破 |
| MFA events | 发现 MFA 疲劳攻击和重置滥用 |
| Permission changes | 发现权限扩张 |
| Group membership changes | 发现隐蔽授权 |
| OAuth consent logs | 发现恶意应用授权 |
| Token usage logs | 发现会话劫持 |

身份日志必须统一用户 ID、设备 ID、IP、地理位置、ASN、认证方式和结果字段，否则跨系统关联会非常困难。

## 7. 定期复核

权限复核要避免变成形式化点击。

有效复核应该给审批人这些上下文：

- 账号是谁。
- 权限能访问什么。
- 最近 30/90 天是否使用。
- 是否有替代低权限角色。
- 如果保留，业务理由是什么。

高风险权限建议按月复核，普通权限按季度或半年复核。长期未使用权限应默认回收。

## 8. 应急账号

Break-glass 账号也要完整审计。

应急账号要求：

- 数量极少。
- 凭证离线保管。
- 使用后强制轮换。
- 登录立即告警。
- 定期演练可用性。
- 不参与普通 SSO 策略变更。

应急账号用于身份平台异常时救场，不能用来绕过日常管控。

## 9. 成熟度指标

建议跟踪：

| 指标 | 目标 |
| --- | --- |
| 管理员抗钓鱼 MFA 覆盖率 | 100% |
| 孤儿账号数量 | 持续下降 |
| 长期未使用高权限账号 | 接近 0 |
| 服务账号 owner 覆盖率 | 100% |
| 权限到期自动回收比例 | 持续提升 |
| OAuth 高风险授权数量 | 可解释且受控 |

身份安全要让每一次授权都可解释、可审计、可撤销。
