# AI System Security Notes

AI 系统安全不能只看模型本身。真实系统通常由模型、检索、工具调用、权限、数据管道、日志和人工流程组成。攻击面也从传统输入输出扩展到了提示词、上下文、工具链和训练数据。

## 1. AI 系统攻击面

| 组件 | 风险 |
| --- | --- |
| Prompt | 指令覆盖、越权诱导 |
| Retrieval | 检索投毒、上下文污染 |
| Tool use | 越权调用、参数注入 |
| Memory | 持久化污染、隐私泄漏 |
| Training data | 数据投毒、后门 |
| Model output | 幻觉、敏感信息泄漏 |
| Logs | 隐私和密钥暴露 |
| Plugins / agents | 权限扩张、供应链风险 |

## 2. 基本安全原则

1. 模型输出不等于可信事实。
2. 模型意图不等于用户授权。
3. Prompt 不能作为安全边界。
4. 工具调用必须有独立权限控制。
5. 检索内容必须视为不可信输入。
6. 高风险动作必须可审计、可回滚。

## 3. Prompt Injection

Prompt injection 的本质是非信任内容影响了模型决策。

常见场景：

- 网页内容中包含“忽略之前指令”。
- 文档中嵌入恶意操作要求。
- 检索结果诱导模型调用工具。
- 用户输入伪装成系统消息。

防护重点：

- 区分 system、developer、user、retrieved content。
- 对检索内容加标签，不让它变成指令。
- 工具调用前做策略检查。
- 高风险操作要求显式确认。
- 记录模型为什么调用工具。

## 4. RAG 安全

RAG 系统的关键风险是“检索到的内容污染决策”。

建议：

- 文档来源分级。
- 检索结果带来源和时间。
- 对高风险答案要求多源交叉验证。
- 不让低信任文档触发高权限工具。
- 对用户上传文档做隔离索引。
- 删除或过期机制明确。

RAG 输出应尽量包含：

```text
answer
sources
confidence
limitations
```

## 5. Tool Use 安全

工具调用是 AI Agent 最大的实际风险点。

每个工具要定义：

| 字段 | 说明 |
| --- | --- |
| Capability | 工具能做什么 |
| Required permission | 调用所需权限 |
| Input schema | 参数结构 |
| Validation | 参数校验 |
| Side effects | 是否会改数据 |
| Audit log | 审计记录 |
| Rollback | 是否可回滚 |

工具调用策略：

- Read-only 工具默认低风险。
- Write / delete / deploy 工具高风险。
- 网络访问工具需限制域名和协议。
- 文件工具需限制路径。
- Shell 工具需最小化命令能力。

## 6. 权限设计

不要让模型继承用户的全部权限。

建议采用：

```text
user permission
model capability
tool policy
runtime guardrail
approval gate
```

高风险动作应该同时满足：

- 用户有权限。
- 工具策略允许。
- 参数通过验证。
- 操作在范围内。
- 有审计记录。
- 必要时有人确认。

## 7. 数据保护

AI 系统容易在以下位置泄漏数据：

- Prompt。
- 检索上下文。
- 模型输出。
- 调试日志。
- 训练样本。
- Memory。
- 第三方工具请求。

基本要求：

- PII 和 secret 检测。
- 日志脱敏。
- 数据最小化。
- 明确保留期。
- 用户隔离。
- 训练数据授权。

## 8. 对抗样本与鲁棒性

对于视觉、语音、分类模型，需要考虑：

- 输入扰动。
- 物理世界对抗样本。
- 分布外输入。
- 置信度校准。
- 模型漂移。

防护：

- 输入质量检查。
- 多模型交叉验证。
- 对抗训练。
- 置信度阈值。
- 人工复核高风险结果。

## 9. 监控与检测

AI 系统需要专门的安全遥测。

建议记录：

- 用户请求类型。
- 检索来源。
- 工具调用参数。
- 工具调用结果。
- 模型拒绝原因。
- 高风险动作审批。
- 异常输出模式。

检测示例：

| 行为 | 风险 |
| --- | --- |
| 短时间大量探测系统提示词 | Prompt extraction |
| 检索结果频繁触发工具调用 | RAG injection |
| 模型尝试访问越权工具 | Policy bypass |
| 输出中出现 secret 格式 | Data leakage |
| 用户反复要求忽略规则 | Injection attempt |

## 10. 安全测试清单

上线前测试：

- Prompt injection。
- 越权工具调用。
- 文件路径穿越。
- SSRF 类输入。
- 敏感信息输出。
- RAG 文档投毒。
- 错误来源引用。
- 高风险操作绕确认。
- 多租户数据串读。
- 日志泄漏。

## 11. 事故响应

AI 事故响应要能回答：

1. 哪个用户触发？
2. 模型看到了什么上下文？
3. 检索到了哪些文档？
4. 调用了哪些工具？
5. 工具产生了什么副作用？
6. 是否泄漏数据？
7. 是否需要撤销操作？

因此，审计链路必须覆盖完整推理和工具调用过程。

## 12. 核心原则

AI 系统安全核心要覆盖：

- 权限在模型外部控制。
- 工具调用可验证。
- 检索内容不可信。
- 高风险动作可审计。
- 数据最小化。
- 失败时安全降级。

Prompt 可以指导模型，但不能替代安全边界。
