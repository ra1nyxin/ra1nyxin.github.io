# Cloud Incident Response Runbook

云事件响应和传统主机响应不同。云控制面、身份、临时凭证、自动化、日志延迟和跨账号关系都会影响调查节奏。好的云响应能力依赖提前准备的权限、日志和 runbook。

## 1. 准备工作

必须提前准备：

- 只读调查角色。
- 隔离和禁用权限。
- 云审计日志集中存储。
- 账号和资源 owner 映射。
- 关键资源标签。
- 跨账号查询方式。
- 应急联系人。

事件发生后再申请权限，通常会浪费最关键的时间。

## 2. 首要问题

云事件开始时先回答：

- 哪个账号或项目受影响。
- 哪个身份执行了异常操作。
- 凭证类型是什么。
- 资源是否被创建、修改或删除。
- 是否访问了敏感数据。
- 是否存在持久化。
- 是否影响其他账号。

调查应从控制面日志开始，再下钻到具体实例。

## 3. 常见事件类型

| 类型 | 关注点 |
| --- | --- |
| Access key leak | 调用历史、权限范围、轮换 |
| Privilege escalation | IAM 变更和新权限 |
| Public storage | 访问日志和数据范围 |
| Crypto mining | 新实例、异常计费、镜像来源 |
| Control plane abuse | API 调用链和自动化身份 |
| CI/CD compromise | 发布凭证和制品完整性 |

不同事件需要不同证据，不要套用单一主机处置流程。

## 4. 遏制动作

可能动作：

- 禁用或轮换访问密钥。
- 撤销会话或临时凭证。
- 隔离实例安全组。
- 冻结可疑 IAM 角色。
- 阻断公开存储访问。
- 暂停可疑自动化流水线。
- 保留磁盘快照和日志。

遏制前尽量确认业务影响和证据保全需求。

## 5. 证据保全

保留：

- 云审计日志。
- IAM 策略和变更历史。
- 资源配置快照。
- 可疑实例磁盘快照。
- 网络流日志。
- 对象存储访问日志。
- CI/CD 执行记录。

证据应复制到隔离账号或受控存储，避免被攻击者删除。

## 6. 持久化检查

检查攻击者是否留下：

- 新 IAM 用户或角色。
- 新访问密钥。
- 后门策略绑定。
- 计划任务或函数。
- 新镜像或启动模板。
- 修改过的安全组。
- 新 webhook 或 CI secret。

只轮换已知泄露密钥可能不够，必须找新增访问路径。

## 7. 恢复

恢复步骤：

- 关闭初始访问路径。
- 轮换相关凭证。
- 回滚异常配置。
- 验证资源完整性。
- 恢复业务服务。
- 增强监控。
- 更新检测规则和基线。

恢复后要继续观察一段时间，确认没有新的异常调用。

## 8. 指标

| 指标 | 目标 |
| --- | --- |
| 云审计日志覆盖 | 100% |
| 调查角色可用性 | 定期验证 |
| 密钥泄露遏制时间 | 持续降低 |
| 跨账号查询时间 | 持续降低 |
| 持久化检查覆盖 | 标准化 |
| 复盘改进关闭率 | 100% |

云响应的关键是控制面优先、身份优先、证据优先。
