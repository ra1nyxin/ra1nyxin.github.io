# Third Party Risk Management

第三方风险不只是供应商问卷。外包团队、SaaS、开源依赖、云服务、支付渠道、数据处理方和托管运维都会引入外部信任关系。治理重点是明确第三方能访问什么、承担什么责任、出现问题时如何收敛影响。

## 1. 第三方分类

| 类型 | 风险 |
| --- | --- |
| SaaS vendor | 数据存储和账号权限 |
| Outsourcing | 人员访问和交付质量 |
| Cloud provider | 基础设施依赖 |
| Open source | 供应链漏洞和维护状态 |
| Data processor | 合规和数据泄露 |
| Managed service | 高权限运维访问 |

不同第三方需要不同控制，不要用同一张问卷解决所有问题。

## 2. 接入前评估

关注：

- 会访问哪些数据。
- 是否接入 SSO。
- 是否需要管理员权限。
- 是否有审计日志。
- 是否支持最小权限。
- 数据是否出境或共享给子处理方。
- 事件通知 SLA。
- 退出和删除机制。

风险评估结论要影响接入方式，而不是只作为采购附件。

## 3. 合同控制

合同应明确：

- 数据处理责任。
- 安全控制要求。
- 漏洞和事件通知时限。
- 审计权。
- 子处理方披露。
- 数据删除和返回。
- 服务终止后的义务。

合同不能替代技术控制，但能定义责任边界。

## 4. 访问治理

第三方访问要求：

- 使用单独身份，不共享账号。
- 最小权限。
- 有过期时间。
- 高风险访问启用 MFA。
- 远程访问通过受控通道。
- 管理操作有日志。
- 定期复核权限。

第三方账号不应成为永久内部员工账号。

## 5. 持续监控

接入后要持续看：

- 权限是否扩大。
- 数据访问是否异常。
- 供应商安全公告。
- 证书和审计报告是否过期。
- 合同范围是否变化。
- 子处理方是否新增。

第三方风险会随业务使用方式变化而变化。

## 6. 开源依赖

开源风险治理：

- 锁定版本。
- 关注维护活跃度。
- 评估许可证。
- 扫描已知漏洞。
- 监控新发布的恶意包。
- 对关键依赖建立替代方案。

不要把开源依赖只归类为开发问题，它也是供应链风险。

## 7. 退出策略

退出时需要：

- 回收账号和 token。
- 导出必要数据。
- 确认数据删除。
- 停止 webhook 和集成。
- 关闭网络访问。
- 保存审计记录。

供应商停用后，遗留 token 和 webhook 是常见风险。

## 8. 风险分级

建议按以下因素分级：

- 数据敏感度。
- 权限等级。
- 业务关键性。
- 网络连通性。
- 替代难度。
- 历史安全表现。

高风险第三方需要更强审计和更频繁复核。

## 9. 指标

| 指标 | 目标 |
| --- | --- |
| 高风险第三方清单 | 完整 |
| SSO 接入率 | 高风险优先 |
| 第三方权限超期 | 0 |
| 合同安全条款覆盖 | 100% |
| 审计日志可用率 | 高风险 100% |
| 退出清理完成率 | 100% |

第三方风险治理的核心是把外部信任关系变成可见、可控、可撤销的工程对象。
