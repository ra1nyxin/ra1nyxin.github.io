# Security Logging Architecture

日志质量比数量更重要。安全日志架构的目标是用可承受的成本收集足够可靠的数据，让检测、调查、审计和复盘都能站得住。

很多安全体系失败时，问题常出在日志字段不稳定、时间不准、上下文缺失、保留期不足，最后只能在事件中临时猜测。

## 1. 设计原则

| 原则 | 说明 |
| --- | --- |
| Purpose-driven | 每类日志都要说明用途 |
| Normalized | 字段命名和语义一致 |
| Time-aligned | 统一时区和时间精度 |
| Tamper-resistant | 关键日志不能被攻击者轻易删除 |
| Cost-aware | 热存、温存、冷存分层 |
| Privacy-aware | 避免不必要收集敏感数据 |

日志系统不能当数据垃圾桶。每条高成本日志都应该对应检测、调查或合规价值。

## 2. 优先级

| 优先级 | 数据源 | 价值 |
| --- | --- | --- |
| P0 | 身份、云控制面、权限变更 | 发现账号接管和权限扩张 |
| P1 | 终端进程、脚本、网络连接 | 还原执行链 |
| P1 | CI/CD、代码仓库、制品仓库 | 发现供应链风险 |
| P2 | SaaS 审计、数据访问日志 | 发现数据外传 |
| P2 | 网络元数据、DNS、代理 | 发现外联和横向移动 |
| P3 | 应用业务日志 | 发现业务滥用 |

优先把 P0/P1 打扎实，再扩大覆盖面。

## 3. 字段规范

最少应统一这些字段：

| 字段 | 说明 |
| --- | --- |
| event_time | 事件发生时间 |
| ingest_time | 采集入库时间 |
| event_type | 事件类型 |
| actor_id | 发起主体 |
| actor_type | 用户、服务、工作负载 |
| source_ip | 来源 IP |
| device_id | 设备标识 |
| resource_id | 目标资源 |
| action | 行为 |
| outcome | 成功、失败、阻断 |
| request_id | 关联请求 |
| trace_id | 跨系统追踪 |

字段规范越早建立，后续检测工程成本越低。

## 4. 时间问题

日志调查最怕时间错乱。

要求：

- 所有系统使用可靠 NTP。
- 存储层统一 UTC。
- 展示层按用户时区转换。
- 保留原始时间字段。
- 记录采集延迟。

检测规则中要考虑日志延迟，特别是 SaaS 和云审计日志，它们可能会延迟到达。

## 5. 保留策略

| 数据 | 热存 | 冷存 |
| --- | --- | --- |
| 身份与权限日志 | 90 天 | 1 年以上 |
| 云控制面日志 | 90 天 | 1 年以上 |
| 终端高价值遥测 | 30-90 天 | 6-12 个月 |
| 网络元数据 | 30 天 | 6 个月 |
| 应用业务日志 | 取决于业务风险 | 取决于合规要求 |

保留期要匹配攻击发现周期。很多供应链和账号滥用事件在数周后才被发现。

## 6. 防篡改

关键日志应满足：

- 写入后不可由普通管理员删除。
- 安全团队有独立读取权限。
- 日志存储与生产环境权限隔离。
- 重要 bucket 或索引启用版本控制或 WORM。
- 删除和保留策略变更会告警。

如果攻击者拿到生产管理员权限就能删除所有审计日志，日志系统不能算安全控制。

## 7. 成本治理

高质量日志架构必须控制成本。

建议：

- 原始日志冷存，归一化字段热存。
- 高频低价值字段采样。
- 检测所需字段优先保留。
- 对调试级日志设置短保留。
- 用数据分层区分在线搜索和离线调查。

降低成本不能靠盲目丢日志，而要知道丢掉的是什么能力。

## 8. 可观测性与安全日志的边界

应用可观测性关注稳定性，安全日志关注行为和证据。两者可以共用 trace_id，但不能互相替代。

安全日志需要额外关注：

- 谁做了什么。
- 是否被授权。
- 是否成功。
- 影响了什么资源。
- 有没有数据流出。
- 这个行为是否常见。

## 9. 日志质量指标

| 指标 | 目标 |
| --- | --- |
| 数据源覆盖率 | 高价值系统 100% |
| 字段完整率 | 核心字段接近 100% |
| 采集延迟 | 明确可度量 |
| 解析失败率 | 持续下降 |
| 重复事件比例 | 受控 |
| 检测规则数据依赖满足率 | 可查询 |

日志架构的价值最终体现在调查速度和检测可靠性上，单纯写入量没有意义。
